1 引言

建筑、工程与施工行业的数字化长期被多个成熟单点软件和专有格式体系分割：设计端有 Revit、Tekla、Rhino、SketchUp、ZWCAD 等工具，结构计算和审查端有 PKPM 等专业系统，造价端有广联达和斯维尔等行业软件，施工和运维端又存在 CDE、项目管理、IoT、BMS、数字孪生与档案平台。单点工具的能力很强，但跨阶段、跨角色、跨标准、跨文件格式、跨审批链条的统一工程治理仍然困难。

ArchIToken 的研发目标是建立一个开放工程智能层：它不把某个模型供应商、某个 CAD/BIM 软件或某个封闭格式作为项目身份，而是以 OpenBIM、CDE、模块工作流、AI 门禁、专业合规、审计和私有化部署为核心。其基本公式为：

ArchIToken = AEC AI-Native + Harness Engineering + OpenBIM CDE Workflow OS

因此，本文的任务不是撰写市场介绍，而是用论文综述的方式，依据当前仓库证据回答四个问题：第一，ArchIToken 到底是什么、边界在哪里；第二，当前系统架构与模块体系如何组织；第三，前端、后端、Agent、openBIM、CAD、数据库和合规底座已经做到什么程度；第四，下一阶段应优先解决哪些生产化缺口。

3 项目定位与边界

3.1 正向定位

ArchIToken 是 AEC 行业 AI-Native 平台、Harness Engineering 系统、OpenBIM CDE Workflow OS、Speckle CDE 互操作运行时、IFCDB-Agent 数据库/Agent 路由和后端原生文件运行时。它的核心价值不是单点建模、单点造价或单点大屏，而是将工程文件、模型、标准、业务对象、审批、AI、工具、权限、审计和交付物组织为一个可追踪、可回滚、可私有化部署的工作流操作系统。

该定位决定了系统的建设重点：统一模块工作台、文件系统、生命周期状态机、审批流、审计事件、OpenBIM 语义、后端原生文件运行时、AI 门禁链、Router/Registry 抽象、多租户权限隔离和可部署基础设施。

3.2 负向边界

ArchIToken 明确不做 Revit、Tekla、PKPM、广联达、斯维尔、中望或 Siemens Building X 的克隆，也不把前端派生文件、截图、水印 PDF、空 Canvas 或假 manifest 当成真实格式支持。私有或复杂格式如 RVT、DWG、DGN、Tekla、Navisworks、Office、PKPM、广联达等，必须通过后端 worker、授权适配器、企业服务或外部进程进入系统，并保留真实源文件绑定、权限、审计和回滚边界。

因此，ArchIToken 的差异化不是“全面替代成熟大厂”，而是在开放数据、AI 可控、私有化交付、证据链和模块闭环上形成系统能力。

3.3 合规边界

专业合规是项目底线。平台每个模块、术语、业务逻辑和 AI 输出都必须绑定专业角色、监管体系、标准版本、条文来源、证据材料、责任人、审批人和输出状态。缺少来源时，系统只能输出“经验建议”，不得输出“合规/不合规/可施工/可报审/可验收/可清关/可入账/可支付/可发布”等结论。

4.2 关键架构原则

  • Registry 替代 Enum：业务模块、模型、工具、渲染器、Schema、规则、工作流等可扩展对象不得被硬编码为不可回滚的 enum 边界。
  • Router 替代直连：模型、推理、工具、工作流、几何、渲染和存储必须通过内部 Router 或 adapter；外部供应商只是运行时选项。
  • OpenBIM 作为语义真源：IFC、IDM、bSDD、BCF、IDS、buildingSMART Validate 和 OpenCDE API 是 CAD/BIM/CIM/GIS 数据流的标准锚点。
  • AI 门禁分离：Generator 不得自评，必须经过 Planner、Generator、Evaluator、RuleChecker、SchemaValidator、Approver。
  • 后端原生文件运行时：复杂工程格式优先读取源文件字节、实体、属性、B-Rep、图层、语义和关系，再生成轻量派生缓存。
  • 可私有化部署：正式路线面向 k8s、Docker、Docker Compose、本地私有化、GPU 节点和弱网/离线环境。

6.1 前端进度

前端已经形成统一模块工作台雏形。03-frontend/app/app/modules/page.tsx 提供 14 模块总入口，03-frontend/app/app/modules/[moduleId]/page.tsx 提供模块详情路由，BusinessModuleWorkbench 与 ModuleWorkbenchShell 负责统一 Shell、模块导航、搜索、主题切换、审计抽屉和 AI 浮窗。

7 关键能力专题分析

7.1 统一 Open CDE 模块工作台

工作台是当前项目最重要的前端成果。它将模块导航、文件、目录、预览、属性、生命周期、审批、AI 建议、审计和业务操作统一到一个 Shell 中。该设计符合“模块不是孤立 landing page 或大屏”的约束，也为后端 HTTP adapter 替换会话态提供了结构基础。

数字孪生模块尤其体现该原则：它不再使用独立 /app/digital-twin 产品入口，而是统一进入 /app/modules/digital_twin，共享 CDE 文件工作台、审批、生命周期、操作队列和 AI 助手。

7.2 OpenBIM 与 CAD/BIM 文件运行时

openBIM 标准基线已明确 IFC、IDM、bSDD、BCF、IDS、buildingSMART Validate、OpenCDE API 和 IFCDB-Agent 的角色。后端 openbim.rs 已提供 IFC ingest、模型记录、viewer manifest、重钢 BOM export 等合同。CAD pipeline 则规定 DXF、STEP、IGES、STL、OBJ、3MF、glTF 等开放格式路线，以及 DWG 必须通过授权或隔离 adapter。

当前重点不应是声明“支持一切格式”，而是为每个格式建立真实源文件、实体图、属性、几何、派生缓存、adapter 证据和失败状态。尤其 DWG、RVT、Tekla、Navisworks、Office 等复杂格式，必须避免用截图、PDF、水印预览或假成功替代真实处理。

7.3 SJG 157-2024 语义字典

SJG 157-2024 被纳入 standard_library，不是独立页面。该集成将深圳建筑工程信息模型语义字典表达为标准元数据、分类树、RDF terms、分类映射、术语投影、项目采用和 BIM model-unit semantic binding。导入脚本使用授权 PDF，通过 pdftotext -layout 提取文本，记录源文件 SHA-256 和 source line，符合“标准来源可追溯”的要求。

7.4 AI Center 管理能力

AI Center 正在从概念页走向管理对象。后端新增接口合同、数据库绑定和可视化面板表；前端 API client 增加相应类型和请求能力；Gateway 已提供读取和更新路由。该方向有助于把 AI、API、RAG、MCP、Agent、模型路由、权限和可视化面板纳入统一管理。

7.5 Project Planning Studio

计划管理模块正在获得独立深度。前端 ProjectPlanningStudio 和 project-planning-studio.ts 扩展甘特、WBS、PERT、RACI、看板、风险矩阵、资源负荷、Mermaid/BPMN 导出等能力；后端迁移补充进度反馈、计划预警和计划调整表。该模块是把市场、设计、采购、生产、施工和财务连接起来的关键治理层。

后端 API 合同 合同成形 Gateway 路由覆盖模块、文件、生成、openBIM、AI Center、语义字典等。 需完整接入 PostgreSQL、对象存储、队列、模型 provider 和 telemetry。
Agent 编排 基础成形 14 模块 prompt 和 planner/generator/evaluator 三段图可运行。 需补确定性 RuleChecker、SchemaValidator、Approver 和审计持久化。
OpenBIM/CAD 标准路线清晰 openBIM baseline、IFC ingest、CAD derivative 合同和 adapter 规则存在。 需真实 worker/sidecar build、smoke evidence、失败证据。
专业合规 制度底座 专业角色、标准来源、AI 状态和规则治理已文档化。 需进入 schema、UI、API、规则库和审批流强制执行。
测试与生产验收 不足 已有单元/E2E/Agent tests，但本次未执行完整矩阵。 需建立每次变更的全量验证报告。

9 主要问题与风险

9.1 前端会话态与后端事实源不一致

当前前端工作台已经提供较完整的文件、审批、生命周期和审计交互，但大量能力由 typed session adapter 驱动。若不尽快切到真实 Gateway，前端状态可能与数据库、对象存储、审计事件和项目事实不一致。

9.2 Worker 与 adapter 证据不足

openBIM/CAD 路线已经清晰，但“真实格式支持”必须由真实 worker、sidecar、CLI、授权 adapter、artifact、manifest、smoke evidence 和失败证据支撑。没有 ODA、LibreDWG、IfcOpenShell、OCCT、FreeCAD、IFCDB-Agent 等证据时，只能标记为 pending/blocked，不能标记 ready。

9.3 专业规则尚未充分运行时化

专业合规文档已经很完整，但规则字段、术语表、标准来源、AI 输出状态、人工复核和签审策略需要进一步落到数据库、API、前端 UI 和 Agent 输出结构中。否则容易出现“文档很严，运行时未强制”的断层。

9.4 依赖版本与许可证边界仍需清理

宪法要求生产依赖补丁级钉住，并禁止 GPL/AGPL/SSPL/BUSL 等进入分发核心。当前前端 package.json 可见若干 ^ 范围依赖，后续需要按生产策略清理锁定并做许可证隔离审查。

9.5 工作区变更规模较大

当前工作区存在大量已修改和未跟踪文件，涉及前端工作台、后端网关、迁移和文档。若不拆分提交和验证，后续回滚、审查和定位会变得困难。

11 结论

综合仓库文档和代码证据，ArchIToken 已经建立了较清晰的系统定位、工程宪法、14 模块 registry、统一模块工作台、Rust Gateway API 合同、Python Agent 三段编排、openBIM 标准基线、CAD derivative 规则、SJG 157 语义字典底座和 AI Center 管理对象。项目当前不再只是概念设计，而是进入了“平台壳、API 面、标准底座、AI 门禁和业务模块深度能力并行推进”的工程化阶段。

但从生产标准看，当前仍属于“试点工程化 + 合同成形 + 局部能力可演示”阶段。关键缺口集中在真实后端持久化、真实 worker/sidecar 证据、前端会话态替换、专业规则运行时强制、AI 门禁全链路审计、完整测试矩阵和依赖许可证清理。下一阶段应优先完成 P0 级稳定化和前后端真实数据闭环，再逐步推进 openBIM/CAD worker 证据、专业合规运行时和生产验收体系。

本文不将当前系统结论表述为“可施工、可报审、可验收、生产完成”。在缺少注册专业人员复核、标准条文来源、真实验证 artifact、签审记录和生产环境测试前，所有 AI 和工程输出均应视为辅助草稿或预审结果。
