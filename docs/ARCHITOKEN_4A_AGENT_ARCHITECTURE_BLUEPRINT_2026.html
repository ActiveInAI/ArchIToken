ArchIToken 4A 与 Agent 企业运营架构蓝图

版本：2026-06-08 · 输入来源：本次会议摘要 + 仓库真源文档。本文用于把战略、4A 架构、BOM 数据底座、Agent 编排、现场移动端和硬件预算落成可执行架构，不替代专业合规、财务审计、等保测评或项目交付签审。

总目标：把公司从“人驱动流程、AI 辅助局部任务”升级为“AI 运行企业经营管理体系，人负责外部关系、创新判断、关键审批和战略思考”。落地方式不是依赖单个大模型，而是以 BOM / Ontology / CDE / 审计为数据底座，以 Agent 编排、Router、RuleChecker、SchemaValidator 和 Approver 形成可控的企业运行系统。
边界：AI 可以采集、整理、生成、比对、提醒和提出建议；客户承诺、供应商谈判、专业签审、财务付款、生产放行、施工验收、对外发布和合规结论必须有人类责任人或授权审批链确认。

1. AI 企业运营战略

组织原则

  • AI 负责业务信息的采集、结构化、执行提醒、交付物草稿、复核比对和运营看板。
  • 人负责客户与供应商界面、技术创新、组织判断、例外处理、专业签审和最终责任。
  • 避免依赖单一“天才员工”或单一模型，把能力沉淀到流程、数据、Agent 和审计体系。

能力原则

  • 单点 Agent 只能处理任务，流程 Agent 才能串业务，企业级 Agent 才能做经营协同。
  • 所有 Agent 输出必须绑定输入数据、规则来源、模型路线、审核状态和审计证据。
  • 模型只是运行时，企业能力来自 Harness、Router、Schema、Registry 和审批门禁。

落地原则

  • 数据架构优先，从 BOM、客户、供应商、项目、构件、工序和交付证据开始。
  • 应用层遵循“AI 采集，人确认”，减少销售、研发、生产、现场人员重复填表。
  • 先做 40-50 万基础硬件和小模型/外部模型混合路由，再按真实负载扩容。

4. 数据架构

核心BOM 是企业运营的核心事实枢纽，但 BOM 不是孤立表格。它要连接需求、方案、深化模型、构件、材料、供应商、采购、生产工序、质检、包装、发运、现场安装、验收和财务成本。

4.1 企业本体 Ontology

对象域 核心对象 权威来源 唯一性要求 典型关系
客户与项目 客户、联系人、手机号、商机、项目、合同、场地。 市场客服 / CRM / 合同。 手机号、客户主体、项目编号唯一；变更留痕。 客户 -> 项目 -> 需求 -> 合同 -> 交付。
设计与工程 方案、模型、图纸、构件、节点、空间、材料、标准族库。 CDE 源文件、深化设计、标准族库。 构件编码、图纸版本、模型 checksum 唯一。 模型 -> 构件 -> BOM -> 工艺 -> 工单。
BOM 与成本 设计 BOM、采购 BOM、加工 BOM、包装 BOM、安装 BOM、BOQ、报价。 深化设计 + 计量造价 + 生产制造。 BOM 行项目编码、版本、适用阶段和状态唯一。 BOM -> 采购单 / 工单 / 发运单 / 成本归集。
供应链 供应商、物料、价格快照、采购单、批次、库存、物流。 材料物流 / 采购 / 仓储。 供应商主体、物料编码、批次号唯一。 供应商 -> 采购单 -> 批次 -> 项目 -> 成本。
生产与现场 工序、工单、设备、质检点、照片、整改、验收。 生产制造 / 施工管理 / 移动端。 工单、工序记录、照片证据和验收记录不可覆盖。 工单 -> 工序 -> 质检 -> 发运 -> 安装 -> 验收。
组织与权限 员工、岗位、角色、权限、审批、会话、审计。 设置中心 / IAM / JumpServer / 审计日志。 账号、角色授权和审批记录唯一且可追溯。 用户 -> 角色 -> 权限 -> 操作 -> 审计证据。

4.2 数据能力分层

能力 职责 首期技术落点 扩展触发
TransactionStore 业务事务、生命周期、审批、回滚和状态一致性。 PostgreSQL。 高并发商业化后读写拆分。
ObjectStore IFC、DWG、PDF、照片、视频、归档包和大文件。 ZFS NAS + S3 兼容对象存储。 生产云对象存储、跨区复制、对象锁。
VectorStore 标准、历史项目、会议纪要、图纸片段和语义检索。 PostgreSQL pgvector 或 Qdrant adapter。 检索量和 RAG 召回质量需要独立扩展时。
GraphStore Ontology、构件关系、供应链关系、流程依赖。 PostgreSQL adjacency table。 图遍历复杂度明显上升后再拆图数据库。
EventStore 审计事件、异步任务、Agent 运行事件和集成 outbox。 PostgreSQL outbox / audit table。 多系统集成和事件流需要 NATS/Kafka 时。
AnalyticsStore 成本、进度、质量、产能和经营分析。 PostgreSQL materialized views。 报表和 BI 负载影响事务库后拆 ClickHouse。

6. 技术架构

技术层 架构选择 落地要求
前端工作台 Next.js / React / TypeScript + PanUI + Open CDE 模块工作台。 16 模块统一 Shell，文件、对象、审批、审计、AI 助手同屏协作。
后端核心 Rust Gateway + Python Agent Orchestrator + worker/sidecar。 业务逻辑禁止直连外部模型或数据库产品，必须经过 Router / StorageRouter / ModuleBackendAdapter。
Agent 编排 Planner -> Generator -> Evaluator -> RuleChecker -> SchemaValidator -> Approver。 生成和评估分离；每一步产生审计证据；专业结论必须 Approver。
模型路由 ModelRouter + InferenceRouter；本地小模型、国内模型和外部高性能模型混合。 DeepSeek、千问、OpenAI-compatible、Claude Code / Codex 类代码能力只能作为适配器，不能写死进业务逻辑。
数据底座 PostgreSQL、对象存储、ZFS、向量检索、事件 outbox、审计日志。 BOM、客户、项目、构件、工序、照片、审批和审计必须可追溯。
部署运行 Kubernetes / Docker Compose / 本地私有化部署。 首期 40-50 万基础硬件跑 3 台物理 CPU 服务器；AI GPU 服务器单列。
安全运维 JumpServer、MFA、RBAC、VLAN、日志、备份、漏洞整改。 100 用户试点按二级基线；商业化和公网多租户按三级预留。

7. 三层 Agent 编排体系

层级 职责 示例 Agent 输入 输出 门禁
个体智能体 执行单个任务。 会议纪要抽取、BOM 行识别、照片质检、价格比对、合同条款抽取。 文件、照片、表单、数据库对象。 结构化草稿、问题清单、建议。 SchemaValidator + 人工确认。
流程智能体 串联跨部门流程。 需求到方案、方案到 BOM、BOM 到采购、生产到发运、现场验收到复盘。 上游模块交付物、状态、审批意见。 流程任务、下一步动作、异常提醒。 WorkflowRouter + RuleChecker + Approver。
企业级智能体 经营协同和战略辅助。 项目复盘、供应链风险、现金流预警、产能排程、组织能力评估。 项目、财务、供应链、生产、客户满意度数据。 经营建议、风险看板、决策备选项。 经营层审批；不得自动对外承诺或付款。

Technical Review 机制

  1. Planner 拆任务，声明输入、目标、标准、风险和工具。
  2. Generator 生成草稿，不允许直接进入生产状态。
  3. Evaluator 使用独立上下文复核是否满足需求。
  4. RuleChecker 对照标准族库、工艺规则、合同边界和安全规则。
  5. SchemaValidator 校验 JSON Schema、Module Schema、IFC Schema、OpenAPI / AsyncAPI 合同。
  6. Approver 由责任人确认，形成版本、签审和审计证据。

8. BOM 核心数据链路

9. 数据治理、安全与变更管理

单一事实来源

  • 手机号、客户、项目、供应商、物料、构件、BOM 行、工单、发运单、验收记录必须有唯一权威源。
  • 其它模块只引用，不复制成新的事实；展示字段可以冗余，但必须保留 source_id 和 version。
  • AI 抽取的数据先进入草稿或待确认状态，确认后才进入主数据。

权限与变更

  • 按岗位、项目、模块、对象、字段和状态控制读写权限。
  • BOM、合同、报价、工单、质检和验收记录禁止无审计覆盖修改。
  • 关键变更必须有原因、影响分析、审批人、版本差异和回滚方案。
控制项 要求 系统实现 验收证据
身份认证 MFA、角色、岗位、项目授权。 Settings Center / IAM / JumpServer。 账号清单、权限矩阵、登录审计。
操作审计 所有文件、BOM、审批和 Agent 操作可追溯。 AuditTrail / EventStore / 日志归档。 审计事件、会话录像、变更记录。
数据备份 在线快照 + 离线备份 + 恢复演练。 ZFS snapshot / ObjectStore / 离线 MG11。 恢复报告、RPO/RTO 记录。
AI 安全 模型路由、工具沙箱、提示注入防护、输出校验。 ModelRouter / ToolRouter / SchemaValidator。 模型调用日志、工具授权、失败记录。
专业合规 缺少专业/监管来源时只能输出经验建议。 RuleChecker / 标准族库 / Approver。 规则引用、审批意见、签审状态。

10. 实施路线

阶段 周期 建设重点 完成标准
第 0 阶段：架构确认 1-2 周 确认 4A 蓝图、BOM 编码规则、业务流程、角色权限和硬件采购清单。 本蓝图评审通过；硬件预算版定稿；BOM 对象字典初稿完成。
第 1 阶段：数据底座 2-4 周 PostgreSQL、对象存储、ZFS、BOM 主数据、客户/项目/供应商/构件主数据。 需求 -> 方案 -> BOM -> 采购 -> 工单 -> 发运 -> 验收的最小数据链路跑通。
第 2 阶段：Agent 流程 4-8 周 会议纪要抽取、需求包、BOM 草稿、采购计划、工序卡、质检照片检查。 20-30 个高价值 Agent 上线，全部有审计和 Approver 状态。
第 3 阶段：现场闭环 8-12 周 移动端扫码、拍照、工序确认、整改复检、验收归档。 一个真实项目完成从发运到验收的数字化证据闭环。
第 4 阶段：经营驾驶舱 12-24 周 成本、进度、供应链、质量、现金流、复盘和企业级 Agent。 经营层可基于真实数据看风险、成本、质量、产能和客户履约。
