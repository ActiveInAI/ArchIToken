1. 结论

当前设计不是单库方案，而是 capability-first 的多存储 data-plane。

Gateway ready
PostgreSQL 是主干事实库；SeaweedFS S3 保存对象字节；Valkey 保存短期状态； NATS JetStream 负责事件流；ClickHouse 承载时序和分析；Qdrant 是可重建向量索引； GraphStore 当前没有外部图数据库，按规则落到 PostgreSQL 的 public.data_graph_edges 邻接表。

2. 设计原则

业务代码面向 StorageRouter capability，不直接绑定某个数据库产品。

事实来源
PostgreSQL
租户、账号、权限、模块、审计、资产、绑定和 fallback 元数据。
派生索引
可重建
Qdrant、ClickHouse、部分全文/图关系能力不得成为唯一事实来源。
写入安全
只读默认
Database Manager 当前只有 PostgreSQL 行级 CRUD；DDL/清库/迁移走审批。

8. Database Manager 运维边界

当前已经有真实数据库管理器，但不是所有引擎都开放写操作。

能力 当前状态 边界
统一 inventory 已接通 Rust manager 并发探测 PostgreSQL、ClickHouse、Valkey、Qdrant、NATS、S3。
PostgreSQL 行级 CRUD 已接通 支持表清单、分页查行、insert、update、delete。update/delete 需要主键对象。
ClickHouse / Qdrant / Valkey / NATS / S3 写操作 只读 inventory 当前不开放破坏性操作。流删除、key 删除、bucket 删除、collection 删除需要后续审批/审计 gate。
DDL / migration / backup / restore 未开放 按架构要求必须进入变更计划、审计、备份校验和审批流，不能从普通表格按钮直接执行。

11. 核心风险与下一步

这是数据库设计层面的真实状态，不是营销式完成度。

尚未完成： Qdrant collections 为 0，NATS streams 为 0， GraphStore 仍是 PostgreSQL adjacency fallback。它们是服务已接通，不代表业务索引和事件流已经装载完成。
下一步建议： 为每个 capability 补充迁移/重建任务、租户隔离测试、审计事件写入和破坏性操作审批 gate。 Database Manager 后续应优先补 query console、导出、变更计划和备份/恢复视图。
