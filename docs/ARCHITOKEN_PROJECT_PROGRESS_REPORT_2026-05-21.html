一、项目定位

项目是什么

ArchIToken 定位为 AEC 行业的 AI-Native 平台、Harness Engineering 系统和 OpenBIM CDE Workflow OS。它把模型、文件、标准、BIM 语义、业务对象、审批、Agent、工具、审计和交付物组织成可运行、可追踪、可回滚、可私有化部署的工程系统。

项目不是什么

ArchIToken 不复刻 Revit、Tekla、PKPM、广联达、中望或 Siemens Building X 等成熟单点产品；它通过 IFC、IDS、BCF、bSDD、OpenCDE、Speckle、后端原生文件运行时和授权适配器连接这些生态。

核心约束：AI 生成内容必须经过 Planner → Generator → Evaluator → RuleChecker → SchemaValidator → Approver。缺少专业或监管来源时，只能输出经验建议，不得输出“可施工、可报审、可验收、合规、支付完成、归档完成”等结论。

三、开发进度总览

已落地 架构与治理

  • 唯一真源、宪法、定位战略、专业合规、模块规范、模块注册、架构、技术栈文档已形成主干。
  • 核心边界明确：Registry 替代 Enum，Router 替代直连供应商，openBIM 作为语义真源。
  • 数字孪生已被约束为 /app/modules/digital_twin 统一模块工作台，不再走独立产品大屏入口。

已落地 前端工作台

  • Next.js / React / TypeScript 前端已有 /app/modules 和 /app/modules/[moduleId]。
  • ModuleWorkbenchShell、ModuleDetailWorkbench、模块 registry、模块操作、文件系统、生命周期和审批交互已具备会话态运行能力。
  • Planning Studio、Digital Twin 操作面板、Open Engineering Viewer、BIM/Office Viewer、AI Center 管理面板等正在扩展。

已落地 后端网关

  • Rust gateway.rs 已覆盖健康检查、模块、文件、生命周期、审计、生成、openBIM、资产、viewer command、Skill、MCP、知识源、AI 中心和语义字典 API。
  • 模块文件、生成任务、生命周期、审计、运行时上下文、权限和分页等核心服务已有代码实现。
  • Cargo workspace 采用 Rust 1.95.0 / edition 2024，后端依赖大量采用补丁级钉住。

推进中 openBIM / CAD / 文件运行时

  • openBIM baseline 覆盖 IFC、IDM、bSDD、BCF、IDS、buildingSMART Validate、OpenCDE 和 IFCDB-Agent。
  • 后端已有 IFC ingest、viewer manifest、重钢 BOM 抽取合同。
  • 本地 IFC/CAD derivative endpoint、源文件 Range/ETag、LibreDWG/ODA/IfcOpenShell/OCCT/FreeCAD 适配边界正在完善。

推进中 Agent 与 AI 门禁

  • Python agent-orchestrator 以 14 模块 ID 构建图，prompt 目录具备 planner/generator/evaluator 三段。
  • 测试覆盖 14 模块 registry、默认模型别名、verdict 约束和本地 prompt 图运行。
  • RuleChecker、SchemaValidator、Approver 已在平台合同和前端状态中表达，后续需要继续沉淀为确定性规则、Schema 与审批记录。

推进中 数据与合规底座

  • 新增迁移覆盖计划进度反馈/预警/调整、SJG 157-2024 语义字典、AI Center 管理表。
  • SJG 157 采用授权 PDF 导入脚本，记录 SHA-256、source line、IFC mapping 与 RDF URI。
  • 多租户 RLS 已在新增表中体现，生产环境仍需要完整 DB、对象存储、队列、模型 provider 和 telemetry 配置。

五、当前工作区变化

当前工作区不是干净状态。生成本报告前，tracked diff 统计为 62 个文件变更、约 15,714 行新增和 3,501 行删除；另有多项未跟踪文件，主要集中在 AI Center 管理、规划管理、数字孪生 WebGPU 视口、SJG 157 语义字典和 OpenClaw API 入口。

主要变更集中区

  • 03-frontend/components：工作台、工程查看器、数字孪生、计划工作室、AI 中心和业务面板。
  • 03-frontend/lib：模块 registry、文件系统、操作、规划管理、CAD/IFC derivative、API client。
  • 04-backend/harness-core：Gateway、AI Center 管理、语义字典、openBIM、模块文件、生成与生命周期服务。
  • 04-backend/migrations：计划管理、SJG 157、AI Center 管理持久化表。
  • 02-architecture 与 docs：模块工作台、数字孪生、openBIM/CAD 与功能地图同步。

需要注意

  • 未提交改动应先按功能域拆分、验证、再提交，避免把架构文档、前端大改和后端迁移混成不可回滚的大提交。
  • 前端 package.json 仍可见若干 ^ 版本范围，和“生产依赖补丁级钉住”的宪法要求存在后续清理项。
  • 当前报告不把未验证的 worker、sidecar、模型 provider、支付或电子签章能力描述为生产可用。

六、阶段性结论

ArchIToken 已经从“概念和页面展示”进入“统一模块工作台 + 后端网关合同 + openBIM/CAD 文件运行时 + AI 门禁链 + 专业合规治理”的工程化阶段。当前最强的已成形能力是 14 模块 registry、统一 CDE 工作台、模块文件/生命周期/审批/审计交互、Rust Gateway API 面、Agent 三段编排和 openBIM/SJG 157 标准底座。

当前仍不能把整个平台描述为生产完成。关键原因是：前端仍有大量会话态 adapter，后端部分服务需要真实 PostgreSQL/S3/队列/model provider/telemetry 配置，RuleChecker 与 SchemaValidator 需要更多确定性规则和标准来源，CAD/BIM worker 需要真实 sidecar 构建、smoke evidence 与失败证据。
