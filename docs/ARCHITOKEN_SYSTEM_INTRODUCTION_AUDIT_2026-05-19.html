二、系统定位与产品边界

AEC AI-Native

AI 不是聊天插件,而是围绕商机、设计、造价、物流、生产、施工、孪生、档案和企业治理的业务对象、文件、标准、审批和审计运行。

Harness Engineering

模型必须经过 ModelRouter、ToolRouter、WorkflowRouter、Schema Gate、RuleChecker、RollbackGuard、审批和审计。Generator 不能自评。

OpenBIM CDE Workflow OS

以 IFC、IDS、BCF、bSDD、COBie、OpenCDE、文件系统、生命周期事务和交付物归档为工程真源,而不是孤立大屏或单点 CAD/BIM 工具。

对外表述建议: 可以说“对标主流 AEC 数据、协同、造价、施工、孪生平台的关键工作流,在开放数据、AI 审计、私有化部署和模块闭环上形成差异化优势”。不能无证据宣称“全面替代或全面超越 Autodesk、Trimble、Siemens、广联达、PKPM、斯维尔、中望”。

三、系统架构

当前仓库实现的是前端工作台、Rust harness/gateway、Python worker adapters、部署基础设施、SDK 和工程文档真源组成的多层架构。

L7 前端工作台Next.js 16.2.4 + React 19.2.5 + TypeScript + PanUI + Three.js/R3F + CDE 文件工作台 + 14 模块路由。
L6 SDK/APIOpenAPI contract、SDK 生成、前后端 API adapter、模块文件 API、AI provider route。
L5 GatewayRust axum gateway、REST/SSE、运行上下文、RBAC、租户/项目隔离、worker result 接收、生产 profile 校验。
L4 AgentPlanner → Generator → Evaluator → RuleChecker → SchemaValidator → Approver 的工程门禁链,配套 prompt tree 与模块 agent contract。
L3 Harness CoreRust workspace: module registry、module lifecycle、generation jobs、runtime execution、viewer adapter、storage router、rollback guard、skill/MCP/knowledge registry。
L2 Worker RuntimePython workers: CAD、openBIM、GIS、point cloud、Office、PDF、OCR、image、ForgeCAD、Cesium、Speckle、document AI 等 adapter contract。
L1 Data/InfraPostgreSQL/Supabase direction、S3-compatible object store、Valkey/Redis-compatible cache、NATS/Temporal、Qdrant/Vector、OpenTelemetry、Prometheus/Grafana。
L0 部署底座Docker Compose、Dockerfile、Kubernetes manifests、Phase 8 scale manifests、Rainbond 模板、production runbook。

九、存在问题与风险

  • 真实 100k 并发尚未认证 Phase 8.1 的 gate、schema、脚本已经实现,但缺少外部/distributed k6 或云压测产生的真实 100k evidence JSON。当前只能说“认证门禁已实现,执行证据待完成”。
  • 复杂私有格式不能宣称原生全支持 RVT、DWG、DGN、Tekla、Navisworks、Office、PKPM、广联达等必须通过后端 worker、授权适配器、企业服务或隔离 sidecar 接入。不能用截图、空 Canvas、伪 PDF 或假 derivative 冒充真实支持。
  • 生产适配器依赖需要逐项安装和验收 IfcOpenShell、LibreOffice、GDAL/OGR、PDAL、FreeCAD、Blender、Speckle、ForgeCAD、Cesium ion、Stirling-PDF、PaddleOCR、MinerU 等都在 worker policy 中有边界,但真实生产能力要看对应二进制、服务、token、许可证和 smoke evidence。
  • 生产环境外部依赖需要实配 PostgreSQL/PgBouncer、S3 对象存储、NATS JetStream、Temporal、Qdrant、Valkey、OpenTelemetry、Prometheus/Grafana、非开发 auth secret、AI provider route 需要在目标企业环境中配置并验证。
  • 历史文档存在同步问题 部分历史文档仍有 11 模块图、未来目标版本或 aspirational stack 描述。当前真实版本应以 lockfile、Cargo.toml、package.json、active registry 和本报告验证结果为准。
  • 标准库和专业责任链不能由 AI 替代 AI 输出默认是辅助草稿。涉及建筑、结构、安全、造价、施工、监理、验收、合同、税务、财务、人力、数据和网络安全的输出必须绑定标准来源、证据链、专业复核和签审状态。
  • 本地工具环境仍有细节缺口 本地 `python` 命令不存在,可用的是 `python3`;k6 未安装;根目录仍有历史 `.claude/settings.local.json`。这些不影响当前通过的主门禁,但会影响环境一致性和项目身份治理。
  • 上线不是只跑本地测试 本地生产就绪门禁通过不等于企业生产已上线。还需要真实环境部署、域名/证书、备份恢复、权限模型、数据迁移、日志留存、渗透测试、压测证据、回滚演练和客户验收。
